## 2. 执行原则

### 2.1 总体原则

- 以 `skills` 为业务规则基准
- 以现有交互协议为基准
- 以现有工具接口结构为基准
- 以链路清晰稳定为优先目标
- 优先保证子图的可中断、可恢复、可观测
- 优先保证 schema、state、步骤模型稳定，再进入具体实现

### 2.2 关于记忆系统

当前记忆系统也是 mock，可推翻重做。

处理原则：

- 不为了兼容 mock 记忆而扭曲 ticket 子图设计
- 方案必须对所有子图通用，不只服务 ticket
- 子图只依赖“当前链路推进所需的最小上下文”
- 如果现有记忆结构与新链路不和谐，应以链路稳定为优先
- 如涉及主图、后处理、服务历史、记忆提取等联动改动，先提出修改方案再确认

补充强约束：

- 一旦 `router` 判定当前轮为连续服务，主图必须沿用上一轮完整服务记忆
- “一轮服务”定义为一次完整的子图服务，而不是一次用户发言
- `post_process` 只负责某一轮完整服务结束后的后处理，不负责下一轮服务调度

### 2.3 关于 prompt

- `ticket_plan.txt` 如重写，必须保留原始版本用于对照
- 其他 ticket 相关 prompt 允许按新方案重写

### 2.4 关于节点数量

- 默认目标是 3 个核心节点：`plan`、`execute`、`reflect`
- 如有明确收益，可以增加极少量高价值节点
- 除非能明显降低复杂度，否则不为单一 case 新增节点

## 6. 验收方案

验收分为四层：

1. schema 验收
2. 节点级验收
3. 子图链路验收
4. 端到端验收

### 6.1 schema 验收

验收对象：

- `TicketState`
- `TicketPlan`
- `TicketStep`
- `StepResult`

验收标准：

- 能覆盖查询、写入、追问、选择、确认、展示、结束
- 字段职责清晰，无明显重复语义
- 支撑中断恢复
- 支撑自动补全预算和循环保护

### 6.2 节点级验收

#### `plan`

验收点：

- 只输出规划结果，不直接中断
- 能根据 skills 生成合理步骤
- 能识别 scene / mode
- 能识别缺失槽位
- 最小必要上下文注入生效

#### `execute`

验收点：

- 能正确执行所有步骤类型
- 中断 payload 稳定
- resume 后能回到当前步骤
- 用户答非所问时不会直接打乱主链路

#### `reflect`

验收点：

- 规则优先生效
- 临时错误可重试
- 无进展能收口
- 需要重规划时能回到 plan

### 6.3 子图链路验收

必须覆盖以下链路：

#### 链路 1：退货创建

- 用户说“我要退货”
- 系统自动查订单
- 多订单或多商品时触发选择
- 写操作前确认
- 成功创建工单
- 返回工单编号、状态、时效

#### 链路 2：换货创建

- 用户说“我要换货”
- 系统查订单和商品
- 校验库存
- 确认后执行
- 成功收口

#### 链路 3：质量问题

- 用户反馈商品有问题
- 系统查订单
- 缺图片凭证时追问
- 收到凭证后重新规划
- 创建质量工单

#### 链路 4：投诉

- 用户投诉服务或物流
- 系统定位订单或工单
- 收集问题描述
- 确认后创建投诉工单

#### 链路 5：权益申请

- 用户申请升级或权益
- 系统先查等级 / 积分
- 信息充分则继续
- 写操作前确认
- 升级或创建工单

#### 链路 6：工单进度查询

- 用户提供或未提供工单号
- 系统优先查工单
- 无工单号时引导选择工单
- 展示处理进度

#### 链路 7：中断恢复

- 在 `ask_user`
- 在 `select`
- 在 `confirm`

这三类中断都必须验证恢复正确。

#### 链路 8：连续服务

- 上一轮未完成 ticket
- 用户后续补充信息
- 系统继续推进，而不是重新开始

### 6.4 端到端验收

需要从 API 或流式入口验证：

- router 正常进入 ticket 子图
- interrupt 能正常返回给前端
- resume 能恢复当前步骤
- 前端能看到过程事件
- 子图结束后状态清理正确
- `intent_queue` 与 ticket 子图解耦后链路仍然正确

## 7. 风险项专项验收

### 7.1 无限循环风险

必须专项验收：

- 连续 replan 无进展
- 连续 retry 无进展
- 连续自动补全无进展
- 连续追问同一槽位

预期：

- 系统必须稳定收口，不可无限循环

### 7.2 写操作风险

必须专项验收：

- 未确认时不能执行写操作
- 用户取消后不能继续写
- 同一写操作重复恢复不能重复提交

### 7.3 上下文污染风险

必须专项验收：

- 用户画像注入不应影响无关规划
- 连续服务上下文不应污染新服务
- 旧对话不应导致错误复用订单 / 工单对象

### 7.4 记忆系统耦合风险

必须专项验收：

- 当前 mock 记忆缺失时，ticket 主链路仍可运行
- 若记忆接口返回异常，不应打断主链路
- 若后续替换记忆实现，ticket 子图不应大改
