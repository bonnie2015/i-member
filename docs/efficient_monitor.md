## 对“效率提升比”的结论

当前这轮重构的收益主要体现在：

- 架构减法
- 职责清晰
- 交互链路可用
- mock 可控
- 正确率稳定

但**如果只看最近一轮完整回归的首轮耗时**，相对于上一版阶段基线，当前并没有提升，反而有回归：

- `SCRM 不通`：`-29.1%` 的效率提升比（即回归 `29.1%`）
- `SCRM 正常 mock`：`-30.2%` 的效率提升比（即回归 `30.2%`）

因此，当前不能把“首轮效率提升”写成正向结果，只能诚实记录为：

- **正确率和可用性提升**
- **效率暂未提升，仍需继续优化**

## 当前判断

### 已经成立的结果

1. 入口和 ticket 子图减法重构已稳定。
2. `scene_guard` 多轮澄清可用。
3. `planner / executor / reflect` 职责更清楚。
4. 前端所需交互类型已可稳定返回：
   - `select_order`
   - `select_product`
   - `select_ticket`
5. 仓库内 deterministic mock 已可稳定支撑回归和前端联调。

### 仍需继续优化的点

1. `scene_guard(qwen3.5:4b) + planner + executor` 叠加后，首轮耗时仍偏高。
2. 查询类路径仍然最慢。
3. 如果后续要沉淀简历量化结果，建议优先以：
   - 正确率提升
   - 可用性提升
   - 交互链路打通
   作为本阶段结果，而不是先写“效率提升”。
