【你的任务】

基于当前工单技能、当前目标和已获取槽位，先判断当前服务是否仍值得继续规划；若仍可继续，再输出一个 5 步以内的宏观计划。

每个步骤只描述：
- `goal`：这一大步要完成什么
- `completion_signal`：这一大步何时算完成
- `target_slots`：完成后应沉淀到全局状态的稳定信息
- `available_tools`：当前步骤允许使用的真实业务工具

【规划规则】

- 只规划宏观主路径，不要把执行时才决定的追问、重试、交互细节拆成多个步骤。
- 能合并的信息收集、查询或确认，应尽量合并成更少的宏观步骤。
- `completion_signal` 不能只写“查询成功”或“拿到字段”，必须写出这一步真正成立的业务条件。
- 技能中规定、但 executor 无法仅靠稳定槽位或工具返回自行推断的限制，必须写进 `goal` 或 `completion_signal`。
- `target_slots` 只保留后续会复用的稳定信息；候选列表、说明字段、一次性工具细节不要写进去。
- 如果当前用户目标与本技能不匹配，应输出 `plan_status = "impossible"`，并给出明确 `plan_reason`，不要继续生成步骤。
- 所有推断、结论、决策必须能现有可用的信息里找到依据，不可以编造，不可以猜测。

【重新规划】

- 当 `planning_mode = replan` 时，优先重点分析 `failed_step.reason` 和 `failed_step.try_process` 中的信息分析失败原因，重新规划后续步骤。
- 若 `failed_step` 为上一轮失败步骤信息，`failed_step.try_process` 具有第一优先级的参考作用，用于规避已经出现的错误。
- 你必须先判断当前服务是否还有继续规划的意义：
  - 若还能形成新的有效路径，则重新规划步骤，`plan_status = "planned"`。
  - 若问题无法绕开或通过其他步骤代替，则立即判定任务失败，`plan_status = "impossible"`。
  - 若用户已明确表示不再继续当前服务，则任务取消，`plan_status = "cancelled"`。
  - 用户更改需求、取消任务或现有信息下任务无法完成，立即结束，`plan_status = "cancelled"`，不再规划步骤。
- 已确认槽位已覆盖的信息不要重复规划，只有仍缺失的关键信息才继续安排步骤。
- failed_step.try_process 里已经做过的查询，禁止重复规划
- replan 必须体现“相对上次失败步骤的调整”，否则算无效计划

【输出要求】

只输出一个符合下列结构的 JSON：

```json
{{
  "plan_status": "planned",
  "plan_reason": "如果本轮为重新规划，必须说明新步骤如何避免上一轮的错误",
  "slots": {{
    "order_id": "N20260305000012"
  }},
  "expected_slots": ["本轮计划预计需要收集的稳定槽位 key"],
  "steps": [
    {{
      "goal": "宏观目标",
      "completion_signal": "这一步何时算完成",
      "target_slots": ["完成后应沉淀到 slots 的稳定字段"],
      "available_tools": ["当前步骤可用的真实业务工具名"]
    }}
  ]
}}
```

【硬限制】

- 只输出 JSON。
- `plan_status` 只允许为 `planned`、`impossible`、`cancelled`。
- `plan_status = "planned"` 时，`steps` 必须非空。
- `plan_status = "impossible"` 或 `plan_status = "cancelled"` 时，`steps` 必须为空，且 `plan_reason` 必须直接说明原因。
- `steps` 不超过 5 步。
- `available_tools` 只允许填写上方真实工具清单中的名称。
