当前执行起点：current_step_index = {current_step_index}

【你的任务】
输出一个 5 步以内的宏观 TicketPlan。

每个步骤只描述：
- `goal`：这一大步要完成什么
- `completion_signal`：这一大步何时算完成
- `target_slots`：完成后应沉淀到全局状态的稳定信息
- `available_tools`：当前步骤允许使用的真实业务工具

【规划规则】
- 只规划宏观主路径，不要把执行时才决定的追问、重试、交互细节拆成步骤。
- 能合并的信息收集、查询或确认，应尽量合并成更少的宏观步骤。
- `completion_signal` 不能只写“拿到字段”或“查询成功”，必须写出这一步真正成立所需的业务条件。
- skill 中规定、但 executor 无法仅靠稳定槽位或 tool 返回自行推断的业务限制，必须写进 `goal` 或 `completion_signal`。
- `target_slots` 只保留后续会复用的稳定信息；当前步骤临时展示、候选、说明和一次性 tool 细节不要写进去。
- 字段命名、可复用字段范围和含义以当前 skill 中的“标准字段约定”为准；若 skill 已明确约定，不要再自行扩写。
- 重新规划时保留已成功步骤，只改动失败点及其后续必要步骤；不要机械重复已经失败且无新增信息支撑的步骤。
- 若任务不属于当前工单服务，输出 `ticket_scene = "others"`，并在 `current_goal` 说明原因；此时 `steps` 必须为空。

【输出要求】
只输出一个符合下列结构的 JSON：

```json
{{
  "current_goal": "一句话描述当前服务目标；若为 others，则说明无法处理原因",
  "ticket_scene": "refund | change | quality | complain | equity | others",
  "slots": {{
    "order_id": "N20260305000012"
  }},
  "expected_slots": ["本轮计划预计需要收集的稳定槽位 key"],
  "steps": [
    {{
      "id": "step_01",
      "goal": "宏观目标",
      "completion_signal": "这一步何时算完成",
      "target_slots": ["完成后应沉淀到 slots 的稳定字段"],
      "available_tools": ["当前步骤可用的真实业务工具名"],
      "is_success": false,
      "result": {{}}
    }}
  ]
}}
```

【硬限制】
- 只输出 JSON。
- `steps` 不超过 5 步。
- `id` 使用连续格式：`step_01`、`step_02`、`step_03` ...
- `available_tools` 只允许填写上方真实工具清单中的名称。
