【判断原则】

- `is_continuous` 必须是 `true` 或 `false`，表示“当前是否和上一轮服务有连续性，在语义上为上一个服务的延续”，包括但不限于：
  - 没有提出新需求的结束性用语或简短回复（intent 可能为 qa，与上一轮服务不一致，但以语义为准。如：好的、谢谢、哦、行、可以、好、知道了）`is_continuous` 是 `true`。
  - 围绕同一服务目标继续追问，`is_continuous` 是 `true`。
  - 对上一轮同一事项补充结果、补充信息或继续推进，`is_continuous` 是 `true`。
- 以下情况通常不算连续：
  - 当前这轮已经切到新的服务目标
  - 话题已经不是同一个
  - 虽然话题相近，但已经是一次完整的新提问
  - `intent`相同，但处理对象、服务动作或当前目标已经变化
- 不要因为主题相近、还是同一个大类意图，就机械判连续
- 若系统提供的上一轮服务信息为空，`is_continuous=false`
- 你必须基于完整输入判断，不要只看最后一句消息

【生成要求】

- 生成的数据代表了用户服务的总结和重要数据，需要能够清晰记录和保留一个完整服务的关键信息。
- `goal` 表示“本次服务的核心目的”，不要照抄用户原话，不要照抄最终回复，简明扼要，主旨清晰，若 `is_continuous=true` 应整合系统提供的上一轮服务目标为新的目标总结
- `summary` 的生成需要严格按照如下要求：发生了什么、目标是什么、结果是什么，如果没完成，原因是什么
- 若 `is_continuous=true`，需要与上一轮服务的 summary 整合成一条，同时保证关键信息不丢失
- 若 `final_status=failed` 或 `cancelled`，必须明确写出为什么结束、卡在什么地方

【输出格式】

```json
{{
  "is_continuous": true,
  "goal": "一句短语或短句，不能为空",
  "summary": "一段高信息密度摘要，不能为空"
}}
```
