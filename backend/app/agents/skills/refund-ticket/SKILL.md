# 订单退换货

## 字段来源

- 所有请求参数、返回字段、字段命名与字段结构，以当前可用 tool 的定义和 API 文档为唯一来源。
- 本 skill 不重复维护字段词典；规划和执行时需要先查对应 tool 的说明，再决定可引用字段。
- 若后续步骤需要复用前序工具结果，应沿用 tool 中已有字段名，不要自行改名或发明同义字段。

## 使用目标

用于以下场景：

- 发起退货
- 发起换货（仅同款同型号）
- 查询退换货工单进度

## 核心规则

1.. 订单类工单必须包含消费渠道 `source_channel`：
   `wechat` / `app` / `web` / `jd` / `tmall` / `douyin` / `api` / `offline`
2. 写操作（创建工单）前必须确认：
   - 退/换商品
   - 数量
   - 问题类型与描述
3. 多商品订单必须先让用户选具体商品，再继续后续写操作。
4. 质量/破损场景需要凭证图片：至少包含问题部位特写、商品整体、面单/订单关联信息；图片不清晰或缺失时先追问补齐，不进入写操作。
5. `user_id` 不作为工具入参显式传递，由鉴权上下文获取。
6. 避免直接询问内部术语（如 `sku`、内部编码），优先用用户视角表达（商品名称、颜色、尺码、数量）。
7. 查询信息展示遵循最小必要原则：只展示与当前问题相关字段，敏感信息脱敏。
8. 工具失败或无权限时如实说明，不编造结果。

## 退换货流程

1. 确认诉求类型：退货 / 换货 / 进度查询。
2. 获取订单：
   - 有订单号：直接查详情。
   - 无订单号：先查订单列表再让用户确认目标订单。
3. 获取订单商品明细，先展示已查到的订单/商品摘要，再让用户选择具体商品。
4. 若为换货：校验目标商品库存是否充足。
5. 质量/破损场景若缺少凭证图片，先追问补齐，再进入写操作。
6. 收敛必要字段后创建工单（`create_ticket`）。

## 工具使用说明

- 字段名、请求参数和返回结构全部以 tool 定义为准；本 skill 只说明使用时机，不重复列字段。
- `get_user_orders`：用户未提供订单号，需要先定位订单候选时使用。
- `get_order_detail`：已有 `order_id`，需要确认订单明细或商品项时使用。
- `search_product`：换货场景下，需要搜索目标商品候选时使用。
- `get_product_detail`：需要补充目标商品确认信息时使用。
- `get_product_stock`：换货前，需要确认库存是否足够时使用。
- `create_ticket`：退货、换货、质量问题等信息已收敛，可以创建工单时使用。
- `get_ticket`：已有工单号，查询单个工单进度时使用。
- `get_tickets`：未提供工单号，需要查询工单候选列表时使用。

补充约束：

- 订单类写操作前，务必确认 `source_channel`、目标商品、数量和问题描述已收敛。
- 若工具返回的信息已足够，不要重复追问用户。

## 规划要点

1. 用户未提供订单号时，优先 `get_user_orders`，再 `select_order` 或展示订单摘要。
2. 涉及具体商品时，优先 `get_order_detail`；多商品订单必须先选商品，再进入写操作。
3. 换货前要确认目标商品并校验库存；必要时再查 `search_product` / `get_product_detail` / `get_product_stock`。
4. 质量/破损且缺图片时，先 `ask_user` 补凭证，不进入 `create_ticket`。
5. 工单进度查询优先 `get_ticket` / `get_tickets`，不要误规划成创建工单。
6. `create_ticket` 前必须有 `confirm`，确认内容至少覆盖商品、数量、问题描述。

## 查询与隐私

1. 仅返回当前用户本人相关订单/工单；疑似越权时先自查原因，无法解决则回复“user_id 非当前用户，无法查询该信息”。
2. 展示时默认脱敏（手机号、地址、邮箱、证件信息等）。
3. 若无法定位订单/工单或工具查询失败，需向用户提示。
