# 工单查询

## 核心定位

- 本 skill 只处理**已有工单**的查询、进度跟踪与结果说明，不负责新建工单。

## 服务范围

本 skill 用于以下诉求：

- 查询已有工单详情
- 查询工单处理进度
- 查询工单当前状态
- 查询工单预计完成时间

以下情况不属于本 skill：

- 发起新的退货、换货、投诉、权益申请等工单
- 仅咨询某类售后规则，但尚未进入已有工单查询
- 无法判断用户是在查已有工单，还是在发起新诉求

## 核心规则

1. 本 skill 只承接“查询已有工单”的诉求，不应误转成创建新工单。
2. 用户已提供工单号时，优先直接查询单个工单，不要再追问内部分类。
3. 用户未提供工单号，但明确是在查工单时，可通过已有业务对象范围查询候选工单，再让用户确认目标工单。
4. 澄清应站在用户视角，优先引导用户补充：
   - 工单号
   - 关联订单
   - 关联商品
   - 申请内容或问题描述
5. 不要求用户理解内部工单类型、内部编码或系统分类。
6. 查询信息展示遵循最小必要原则，只展示与当前问题相关的信息，敏感信息脱敏。
7. 工具失败、无结果或无权限时如实说明，不编造结果。
8. 服务选择优先按“用户当前要做的动作”判断：
   - 若目标是查看已经存在的处理结果、处理状态、工单详情、售后进度，优先归到本 skill；
   - 若目标是新发起退货、投诉、权益申请或新的售后处理请求，则不归本 skill。

## 处理原则

1. 先判断用户是否明确在查询已有工单。
2. 若已有工单号，优先单工单查询。
3. 若无工单号但查询意图明确，优先查询候选工单并让用户选择。
4. 查询结果足够时直接说明，不做无价值追问。
5. 查询不到目标工单时，应明确说明并引导用户补充更具体信息。

## 工具使用时机

- `get_ticket`：已有工单号，查询单个工单详情或进度时使用。
- `get_tickets`：未提供工单号，需要查询候选工单列表并进一步确认目标工单时使用。

补充约束：

- 若工具结果已足够，不要重复让用户补充同样的信息。
- 若当前其实是要发起新售后，而不是查询已有工单，应交回其他对应服务处理。

## 查询与隐私

1. 仅展示当前用户本人相关工单；疑似越权时拒绝并提示“无法查询该信息”。
2. 展示时默认脱敏（手机号、地址、邮箱、证件信息等）。
3. 若无法定位工单或工具查询失败，需向用户明确提示。
