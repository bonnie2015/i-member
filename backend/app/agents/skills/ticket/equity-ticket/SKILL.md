# 会员权益工单

## 核心定位

- 所有请求参数、返回字段、字段命名与字段结构，以当前可用 tool 的定义和 API 文档为唯一来源。
- 本 skill 不重复维护字段词典；执行前需要先查对应 tool 的说明，再决定可引用字段。
- 若后续步骤需要复用前序工具结果，应沿用 tool 中已有字段名，不要自行改名或发明同义字段。
- 当前系统是全渠道统一管理系统；线上线下用户身份与会员服务可能在同一系统内统一管理。

本 skill 用于以下诉求：

- 发起升级申请
- 处理升级异常申诉
- 创建上述两类工单并继续处理申请

## 核心规则

1. 写操作（升级/创建工单）前必须确认。
2. `user_id` 不作为工具入参显式传递，由鉴权上下文获取。
3. 避免直接询问内部术语，用用户可理解话术表达。
4. 查询信息展示遵循最小必要原则：只展示与当前问题相关字段，敏感信息脱敏。
5. 工具失败或无权限时如实说明，不编造结果。

## 服务资格与受理条件

### 升级申请 / 升级异常申诉

- 只有当用户明确表达“我要申请升级”“请帮我发起升级申请”时，才进入升级申请处理。
- 只有当用户明确表达“升级异常需要申诉”“升级异常帮我处理”时，才进入升级异常申诉处理。
- 升级申请或异常申诉前必须先查询当前等级与升级条件；若用户尚不满足条件，应直接说明限制，不强行进入写操作。
- 处理前至少要确认：
  - 当前等级
  - 目标等级或目标权益
  - 申请原因

### 已有工单查询转交

- 用户明确表达“查权益工单/升级申请进度”时，应统一转交已有工单查询型服务处理。
- 已有工单查询不由本 skill 正式承接。

### 转交到 QA 的情况

- 用户只是查询积分余额、积分明细、可用权益、会员制度、升级条件说明。
- 用户只是想了解“还差多少条件”或“等级规则是什么”，并未明确提出“申请升级”或“升级异常申诉”。
- 这类信息查询不属于本 skill 的正式承接范围，应交由 QA 解答。

## 不进入写操作的情况

- 仅咨询积分规则、权益内容、升级条件、会员制度，属于规则/信息查询。
- 查询结果已经足够直接回答用户。
- 用户尚不满足升级或权益申请条件。
- 目标等级、目标权益或申请原因仍不明确。

## 处理原则

1. 先判断当前诉求属于升级申请、升级异常申诉，还是规则咨询 / 已有工单查询。
2. 查询优先：
   - 升级场景先查 `get_user_level`
3. 若当前只是规则咨询或资格说明，不由本 skill 承接。
4. 需要写操作时先确认关键项（目标权益/目标等级/原因）。
5. 若用户不具备相应条件，应直接说明限制，不强行创建工单。

## 场景细分

### query flow

- 本 skill 不承担积分、权益内容或升级条件说明类直接问答。
- 未明确提出“升级申请”或“升级异常申诉”时，不进入本 skill。
- 若是已有工单进度查询，应统一转交已有工单查询型服务。
- 若是规则类咨询，应交由 QA 处理。

### application / appeal flow

- 必须先核实当前等级与升级条件。
- 若用户不满足升级条件，应明确解释原因和差距，而不是继续推进写操作。
- 创建前确认中至少应覆盖：目标等级/目标权益、申请原因、当前条件状态。

## 标准字段约定

以下字段是本场景推荐复用的标准字段名。不同模型在处理本场景时，若需要跨轮次或跨工具复用信息，优先使用这些名字，不要自行发明同义字段。

### 通用可选槽位

- `problem_description`：用户的申请原因、异常说明或诉求摘要；统一使用这个名字，不要产出 `reason`
- `ticket_id`：已创建或已确认的工单号
- `ticket_status`：工单当前状态
- `expected_finish_time`：工单预计完成时间

### query 可选字段

- 查询已有工单时优先：`ticket_id`、`ticket_status`

### upgrade 可选槽位

- 若升级申请需要跨步复用，可使用：`problem_description`
- 创建或查询后补：`ticket_id`、`ticket_status`、`expected_finish_time`

## 工具使用说明

- 字段名、请求参数和返回结构全部以 tool 定义为准；本 skill 只说明使用时机，不重复列字段。
- 使用本 skill 时，只能从当前系统真实存在的工具中选择，不允许发明或改写工具名。
- `get_user_level`：查询当前等级、下一等级目标或升级条件时使用。
- `create_ticket`：需要创建权益类工单时使用。
- `get_ticket`：已有工单号，查询单个权益工单进度时使用。
- `get_tickets`：未提供工单号，需要先查工单候选列表时使用。

补充约束：

- 升级或创建工单前必须先确认目标等级、目标权益或具体原因。
- 如果查询结果已足够回答用户，不要机械进入写操作。
- 若当前属于规则咨询或资格不满足，应先解释限制，不要机械推进到写操作。
- 不存在独立的“权益资格校验工具”或“升级工单专用工具”；资格判断应基于现有查询结果和业务规则完成，不要虚构额外工具。

## 查询与隐私

1. 仅返回当前用户本人相关信息；疑似越权时拒绝并提示“无法查询该信息”。
2. 展示时默认脱敏（手机号、地址、邮箱、证件信息等）。
3. 若工具查询失败，统一合规话术：
   “无法查询到相关信息，请稍后重试；如需继续处理，请补充必要信息。”
