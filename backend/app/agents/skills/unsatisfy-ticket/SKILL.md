# 质量问题与投诉

## 字段来源

- 所有请求参数、返回字段、字段命名与字段结构，以当前可用 tool 的定义和 API 文档为唯一来源。
- 本 skill 不重复维护字段词典；规划和执行时需要先查对应 tool 的说明，再决定可引用字段。
- 若后续步骤需要复用前序工具结果，应沿用 tool 中已有字段名，不要自行改名或发明同义字段。

## 使用目标

用于以下场景：

- 质量问题（破损/瑕疵/功能异常）
- 服务投诉（客服、物流、体验等）
- 查询质量/投诉工单进度

## 核心规则

1. 优先查询，不先追问：能通过工具拿到的信息不要先问用户。
2. 订单类工单必须包含消费渠道 `source_channel`：
   `wechat` / `app` / `web` / `jd` / `tmall` / `douyin` / `api` / `offline`
3. 写操作（创建工单）前必须确认：
   - 问题类型（`quality` 或 `complain`）
   - 关联订单/商品（如有）
   - 问题描述与诉求
4. 多商品订单必须先让用户选具体商品，再继续后续写操作。
5. 质量问题必须提供凭证图片：至少包含问题部位特写、商品整体、面单/订单关联信息；图片不清晰或缺失时先追问补齐，不进入写操作。
6. `user_id` 不作为工具入参显式传递，由鉴权上下文获取。
7. 避免直接询问内部术语（如 `sku`、内部编码），优先用用户视角表达（商品名称、颜色、尺码、数量）。
8. 查询信息展示遵循最小必要原则：只展示与当前问题相关字段，敏感信息脱敏。
9. 工具失败或无权限时如实说明，不编造结果。

## 处理流程

1. 确认诉求类型：质量问题 / 投诉 / 工单进度查询。
2. 获取关联信息：
   - 有订单号：直接查订单详情。
   - 无订单号：先查订单列表再让用户确认目标订单。
3. 先展示已查到的订单/商品摘要，再让用户补充或选择具体对象。
4. 质量问题若缺凭证图片，先追问补齐，再进入写操作。
5. 收敛必要字段后创建工单（`create_ticket`，`ticket_type` 为 `quality` 或 `complain`）。
6. 返回工单号、状态、预计处理时效与后续查询方式。

## 场景细分

### quality flow

- 必须优先确认是否存在关联订单或商品。
- 必须判断是否缺少凭证图片。
- 若缺少问题部位特写、商品整体或订单关联信息，不进入写操作。
- 创建前确认中必须明确问题描述、关联商品和附件情况。

### complain flow

- 可存在无商品破损凭证的投诉场景，但应尽量定位订单、服务节点或物流节点。
- 应优先收敛投诉对象、发生经过、用户诉求。
- 若无法自动定位关联订单，可通过时间范围、渠道、问题描述继续收敛。
- 创建前确认中必须明确投诉类型、经历摘要和用户期待结果。

### query flow

- 查询进度优先走 `get_ticket` / `get_tickets`。
- 若用户未提供工单号，优先查工单列表并引导选择，而不是直接重新创建工单。

## Write Constraints

- `quality` 与 `complain` 必须显式区分，不得混用 `ticket_type`。
- 写前确认必须包含当前工单类型、关联对象、问题描述和下一步处理预期。
- 订单类工单必须带 `source_channel`。

## 工具使用说明

- 字段名、请求参数和返回结构全部以 tool 定义为准；本 skill 只说明使用时机，不重复列字段。
- `get_user_orders`：未提供订单号，需要先定位关联订单时使用。
- `get_order_detail`：已有 `order_id`，需要确认订单和商品明细时使用。
- `get_product_detail`：质量问题场景下，需要补充商品确认信息时使用。
- `create_ticket`：`quality` 或 `complain` 信息已收敛，可以提交工单时使用。
- `get_ticket`：已有工单号，查询单个工单进度时使用。
- `get_tickets`：未提供工单号，需要先查询工单候选列表时使用。

补充约束：

- 订单类工单写前必须保证 `source_channel` 已明确。
- 质量问题缺少凭证图片时，不进入 `create_ticket`。
- 查询能解决的问题优先查询，不要过早进入写操作。

## 规划要点

1. 先判断当前是 `quality` 还是 `complain`；若用户诉求是查询已有工单进度，应作为对应场景下的查询型诉求处理，不单独抽成新的 ticket_scene。
2. 无订单号时优先先查订单列表，再引导选择目标订单。
3. 质量问题必须检查是否缺少凭证图片；缺图时先 `ask_user`，不要进入写操作。
4. 投诉场景可无图片，但要优先收敛投诉对象、发生经过、时间范围和用户诉求。
5. 查询进度优先 `get_ticket` / `get_tickets`，不要把查询误规划成新建工单。
6. `create_ticket` 前必须有 `confirm`，确认内容至少覆盖工单类型、关联对象、问题描述或投诉摘要。

## 查询与隐私

1. 仅返回当前用户本人相关订单/工单；疑似越权时拒绝并提示“无法查询该信息”。
2. 展示时默认脱敏（手机号、地址、邮箱、证件信息等）。
