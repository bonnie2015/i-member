# 质量问题与投诉

## 核心定位

- 所有请求参数、返回字段、字段命名与字段结构，以当前可用 tool 的定义和 API 文档为唯一来源。
- 本 skill 不重复维护字段词典；执行前需要先查对应 tool 的说明，再决定可引用字段。
- 若后续步骤需要复用前序工具结果，应沿用 tool 中已有字段名，不要自行改名或发明同义字段。

本 skill 用于以下诉求：

- 质量问题（破损/瑕疵/功能异常）
- 服务投诉（客服、物流、体验等）
- 查询质量/投诉工单进度

## 核心规则

1. 优先查询，不先追问：能通过工具拿到的信息不要先问用户。
2. 订单类工单必须包含消费渠道 `source_channel`：
   `wechat` / `app` / `web` / `jd` / `tmall` / `douyin` / `api` / `offline`
3. 写操作（创建工单）前必须确认：
   - 问题类型（`quality` 或 `complain`）
   - 关联订单/商品（如有）
   - 问题描述与诉求
4. 多商品订单必须先让用户选具体商品，再继续后续写操作。
5. 质量问题必须提供凭证图片：至少包含问题部位特写、商品整体、面单/订单关联信息；图片不清晰或缺失时先追问补齐，不进入写操作。
6. `user_id` 不作为工具入参显式传递，由鉴权上下文获取。
7. 避免直接询问内部术语（如 `sku`、内部编码），优先用用户视角表达（商品名称、颜色、尺码、数量）。
8. 查询信息展示遵循最小必要原则：只展示与当前问题相关字段，敏感信息脱敏。
9. 工具失败或无权限时如实说明，不编造结果。

## 服务资格与受理条件

### 质量问题

- 用户明确表达商品存在破损、瑕疵、功能异常、少件错发等问题时，进入质量问题处理。
- 质量问题受理前，应尽量定位到关联订单与具体商品；无法定位时先继续收敛，不直接建单。
- 质量问题至少要确认：
  - 关联商品或业务对象
  - 问题描述
  - 用户诉求
- 质量问题必须具备凭证图片；若缺少问题部位特写、商品整体或订单关联信息，不进入写操作。

### 投诉

- 用户明确表达对客服、物流、门店、服务态度、处理体验等不满，并希望投诉、反馈、申诉时，进入投诉场景。
- 投诉不一定要求商品破损凭证，但应尽量定位关联订单、服务节点或发生时间范围。
- 投诉至少要确认：
  - 投诉对象
  - 发生经过
  - 用户诉求或期望结果
- 若用户只是表达情绪但未提出处理诉求，应先继续收敛，不要直接建投诉工单。

### 进度查询

- 用户明确表达“查工单进度/查售后进度/查投诉处理情况”时，优先进入已有工单查询路径。
- 已有工单号时优先直接查单个工单，不要求先补 ticket_type、biz_id。
- 无工单号时，优先查当前用户工单列表并让用户确认目标工单；只有在工单列表仍无法定位时，才继续补充订单或商品信息。
- 查询类诉求不应误转成新建工单。

## 不进入写操作的情况

- 仅咨询规则、政策、时效，不属于直接建单。
- 无法定位关联订单/商品/服务节点，且当前信息不足以支撑建单。
- 质量问题缺少必要凭证图片。
- 投诉场景中投诉对象、发生经过、用户诉求仍明显不清楚。
- 当前诉求更适合查询已有工单，而不是重新创建工单。

## 处理原则

1. 先判断当前诉求属于质量问题、投诉还是对应场景下的工单进度查询。
2. 获取关联信息时遵循：
   - 有订单号：直接查订单详情。
   - 无订单号：先查订单列表再让用户确认目标订单。
3. 多商品订单先收敛到具体商品，再进入任何写操作。
4. 质量问题缺凭证图片时，先补齐再进入写操作。
5. 投诉场景优先收敛投诉对象、发生经过和用户诉求。
6. 查询已有工单优先走 `get_ticket` / `get_tickets`，不要误转成新建工单。

## 场景细分

### quality flow

- 必须优先确认是否存在关联订单或商品。
- 必须判断是否缺少凭证图片。
- 若缺少问题部位特写、商品整体或订单关联信息，不进入写操作。
- 创建前确认中必须明确问题描述、关联商品和附件情况。
- 若当前只能证明“有问题”，但无法说明具体问题类型或诉求，先继续收敛，不直接建单。

### complain flow

- 可存在无商品破损凭证的投诉场景，但应尽量定位订单、服务节点或物流节点。
- 应优先收敛投诉对象、发生经过、用户诉求。
- 若无法自动定位关联订单，可通过时间范围、渠道、问题描述继续收敛。
- 创建前确认中必须明确投诉类型、经历摘要和用户期待结果。
- 若投诉对象、经过或诉求仍明显缺失，不进入写操作。

### query flow

- 查询进度优先走 `get_ticket` / `get_tickets`。
- 若用户已提供工单号，直接用 `get_ticket` 查询，不再继续要求 ticket_type / biz_id。
- 若用户未提供工单号，优先查工单列表并引导选择，而不是直接重新创建工单。
- 查询路径的稳定字段优先控制在回答当前问题所必需的信息，例如 `ticket_id`、`ticket_status`；不要预先上浮 `title`、`content`、`biz_id` 等非首要字段。
- 若查询结果已足够回答用户，不要继续扩展到创建流程。

## 标准字段约定

以下字段是本场景推荐复用的标准字段名。不同模型在处理本场景时，若需要跨步骤、跨轮次或跨工具复用信息，优先使用这些名字，不要自行发明同义字段。

### 通用可选槽位

- `order_id`：已确认的关联订单号
- `order_item_id`：已确认的订单商品项 ID
- `product_id`：已确认的关联商品 ID
- `sku_id`：已确认的关联规格 ID
- `source_channel`：订单消费渠道
- `problem_description`：质量问题描述、投诉经过或用户诉求摘要；统一使用这个名字，不要产出 `reason`
- `evidence_images`：用户上传的售后凭证图片，仅指问题凭证，不含商品展示图
- `ticket_id`：已创建或已确认的工单号
- `ticket_status`：工单当前状态
- `expected_finish_time`：工单预计完成时间

### quality 可选槽位

- 优先：`order_id`、`product_id`、`problem_description`
- 质量问题必须补：`evidence_images`
- 创建或查询后补：`ticket_id`、`ticket_status`、`expected_finish_time`

### complain 可选槽位

- 优先：`problem_description`
- 若已定位关联订单/对象，可补：`order_id`、`product_id`
- 创建或查询后补：`ticket_id`、`ticket_status`、`expected_finish_time`

### query 可选字段

- 已有工单号时优先：`ticket_id`、`ticket_status`
- 若需先定位关联对象，可补：`order_id`、`product_id`
- 不要把 `ticket_type`、`biz_id`、标题、描述、候选列表等展示信息当作首要标准字段优先复用

## 写操作约束

- `quality` 与 `complain` 必须显式区分，不得混用 `ticket_type`。
- 写前确认必须包含当前工单类型、关联对象、问题描述和下一步处理预期。
- 订单类工单必须带 `source_channel`。

## 工具使用说明

- 字段名、请求参数和返回结构全部以 tool 定义为准；本 skill 只说明使用时机，不重复列字段。
- 使用本 skill 时，只能从当前系统真实存在的工具中选择，不允许发明或改写工具名。
- `get_user_orders`：未提供订单号，需要先定位关联订单时使用。
- `get_order_detail`：已有 `order_id`，需要确认订单和商品明细时使用。
- `get_product_detail`：质量问题场景下，需要补充商品确认信息时使用。
- `create_ticket`：`quality` 或 `complain` 信息已收敛，可以提交工单时使用。
- `get_ticket`：已有工单号，查询单个工单进度时使用。
- `get_tickets`：未提供工单号，需要先查询工单候选列表时使用。

补充约束：

- 订单类工单写前必须保证 `source_channel` 已明确。
- 质量问题缺少凭证图片时，不进入 `create_ticket`。
- 查询能解决的问题优先查询，不要过早进入写操作。
- 若当前仅属于规则咨询、解释说明或纯进度查询，不要机械推进到 `create_ticket`。
- 不存在独立的“投诉资格校验”“质量问题专用建单”等额外工具；如需判断条件，应基于现有查询结果和业务规则完成，不要虚构工具。

## 查询与隐私

1. 仅返回当前用户本人相关订单/工单；疑似越权时拒绝并提示“无法查询该信息”。
2. 展示时默认脱敏（手机号、地址、邮箱、证件信息等）。
