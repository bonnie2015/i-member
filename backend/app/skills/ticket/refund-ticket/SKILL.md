# 订单退货

## 核心定位

本 skill 用于以下诉求：

- 发起退货
- 查询退货工单进度

## 核心规则

1. 订单类工单必须确认消费渠道。
2. 写操作（创建工单）前必须确认：
   - 退货商品
   - 数量
   - 问题类型与描述
3. 多商品订单必须先让用户选具体商品，再继续后续写操作。
4. 质量/破损场景需要凭证图片：至少包含问题部位特写、商品整体、面单/订单关联信息。
5. 避免直接询问内部术语，优先用用户视角表达（商品名称、颜色、尺码、数量）。
6. 查询信息展示遵循最小必要原则，敏感信息脱敏。
7. 工具失败或无权限时如实说明，不编造结果。

## 受理条件

- 用户明确表达退货/退款意图才进入受理流程，仅咨询政策、时效不视为建单诉求
- 受理前必须定位到具体订单与商品
- 退货至少确认：商品、数量、原因
- 质量/破损场景需要凭证图片，不足时追问补齐，不进入写操作

## 不进入写操作的情况

- 仅咨询规则、时效、政策（先查 policy.md，如需继续再受理）
- 无法定位订单或商品
- 多商品订单未明确目标商品
- 凭证不足

## 处理原则

1. 先判断诉求属于退货还是工单进度查询。
2. 优先查询，不要在可直接查询时先追问。
3. 获取订单时遵循：
   - 有订单号：直接查详情。
   - 无订单号：先查订单列表再让用户确认目标订单。
4. 获取订单商品明细后，先收敛到具体商品，再进入任何写操作。
5. 质量/破损场景缺少凭证图片时，不进入写操作。
6. 若当前问题只是查询工单进度，优先查询已有工单，不要误转成新建工单。

## 场景细分

### refund flow

- 必须先确认用户诉求是“退货/退款”，而不是仅咨询规则。
- 必须先定位订单和具体商品，再进入任何写操作。
- 创建前确认中至少应覆盖：退货商品、数量、退货原因、是否有质量问题凭证。

### query flow

- 查询优先走已有工单，不重建工单。
- 已有工单号时，优先围绕工单号直接查单，不预先要求用户补充非首要信息。
- 无工单号时可通过订单或业务对象范围收敛候选工单。

## 工具使用时机

- `get_user_orders`：用户未提供订单号，需要先定位订单候选时使用，控制搜索数量，一次最多搜索5条。
- `get_product_detail`：需要获取待售后商品其他颜色、尺码等信息时使用。
- `search_products`：需要搜索商品时使用。
- `create_ticket`：退货、质量问题等信息已收敛，可以创建工单时使用。
补充约束：

- 订单类写操作前，务必确认 消费渠道、目标商品、数量和问题描述已收敛。
- 若已查到的信息足够，不要重复追问用户。
- `create_ticket` 前必须完成面向用户的确认，确认内容至少覆盖商品、数量和问题描述。
- 若当前仅属于规则咨询或条件不满足，应先解释限制，不要机械推进到 `create_ticket`。
- **需要确认退货规则、时效、条件等政策信息时，使用 `read_file` 读取 `app/skills/ticket/refund-ticket/policy.md`，不要在不知道的情况下编造政策内容。**

## 查询与隐私

1. 仅展示当前用户本人相关订单/工单；疑似越权时拒绝并提示“无法查询该信息”。
2. 展示时默认脱敏（手机号、地址、邮箱、证件信息等）。
3. 若无法定位订单/工单或工具查询失败，需向用户提示。
