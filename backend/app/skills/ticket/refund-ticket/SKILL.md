# 订单退换货

## 核心定位

本 skill 用于以下诉求：

- 发起退货
- 发起换货（仅同款同型号）
- 查询退换货工单进度

## 核心规则

1. 订单类工单必须确认消费渠道。
2. 写操作（创建工单）前必须确认：
   - 退/换商品
   - 数量
   - 问题类型与描述
3. 多商品订单必须先让用户选具体商品，再继续后续写操作。
4. 质量/破损场景需要凭证图片：至少包含问题部位特写、商品整体、面单/订单关联信息；图片不清晰或缺失时先追问补齐，不进入写操作。
5. 避免直接询问内部术语，优先用用户视角表达（商品名称、颜色、尺码、数量）。
6. 查询信息展示遵循最小必要原则，敏感信息脱敏。
7. 工具失败或无权限时如实说明，不编造结果。

## 服务资格与受理条件

### 退货

- 用户明确表达“要退货/退款/退回商品”时，才进入退货受理流程；若只是咨询退货政策、时效、规则，属于规则咨询，不直接视为建单诉求。
- 退货受理前必须能定位到具体订单与具体商品；若无法定位订单或无法确认商品，不进入写操作。
- 退货至少要确认：
  - 退货商品
  - 数量
  - 退货原因
- 若退货理由属于质量/破损/少件/错发等问题，需要补充凭证图片；凭证不足时，不进入写操作。
- 若用户只表达模糊不满，但尚未明确“要退货”，应先收敛诉求，不要直接按退货建单。

### 换货

- 换货仅适用于订单内原商品的同款同型号换尺码、换颜色、换规格等诉求；若用户想改款、改品类或更换为不同商品，不按换货处理。
- 换货受理前必须能通过订单详情定位到具体订单商品项，以及用户想更换到的目标规格。
- 换货至少要确认：
  - 原商品
  - 目标规格/型号
  - 数量
  - 换货原因
- 换货前必须校验库存；库存不足时，不应直接承诺可换货，应先说明限制并引导其他可行方案。
- 若用户诉求本质上是退货退款，而不是更换同款规格，不应误按换货建单。

### 进度查询

- 用户明确表达“查退货/换货工单进度”“查售后处理情况”时，应优先进入查询已有工单路径。
- 已有工单号时优先 `get_ticket`；无工单号但可定位到订单或业务对象时，优先 `get_tickets`。
- 查询类诉求不应误转成新建工单。

## 不进入写操作的情况

- 仅咨询退换货规则、时效、政策，不属于直接建单。
- 无法定位订单、商品或当前业务对象。
- 多商品订单但尚未明确目标商品。
- 换货场景尚未明确目标规格，或库存未确认。
- 质量/破损导致的退换货，但凭证图片缺失或不清晰。
- 当前信息已明确说明诉求不属于退货/换货，而是其他类型售后，应转到对应场景处理。

## 处理原则

1. 先判断诉求属于退货、换货还是工单进度查询。
2. 优先查询，不要在可直接查询时先追问。
3. 获取订单时遵循：
   - 有订单号：直接查详情。
   - 无订单号：先查订单列表再让用户确认目标订单。
4. 获取订单商品明细后，先收敛到具体商品，再进入任何写操作。
5. 换货场景的关联商品必须来自订单详情；确认目标规格后通过鬼冢虎商品详情校验库存。
6. 质量/破损场景缺少凭证图片时，不进入写操作。
7. 若当前问题只是查询工单进度，优先查询已有工单，不要误转成新建工单。

## 场景细分

### refund flow

- 必须先确认用户诉求是“退货/退款”，而不是仅咨询规则。
- 必须先定位订单和具体商品，再进入任何写操作。
- 创建前确认中至少应覆盖：退货商品、数量、退货原因、是否有质量问题凭证。

### change flow

- 必须先确认是订单内原商品“同款同型号”内的换货，不是改款或改品类。
- 必须确认目标规格，并完成库存校验后，才可进入写操作。
- 创建前确认中至少应覆盖：原商品、目标规格、数量、换货原因、库存状态。

### query flow

- 查询优先走已有工单，不重建工单。
- 已有工单号时，优先围绕工单号直接查单，不预先要求用户补充非首要信息。
- 无工单号时可通过订单或业务对象范围收敛候选工单。

## 工具使用时机

- `get_user_orders`：用户未提供订单号，需要先定位订单候选时使用。
- `get_order_detail`：已有订单号，需要确认订单明细或商品项时使用。
- `onitsuka_get_product_detail`：订单商品信息不足，或换货前需要确认目标规格与库存时使用。
- `create_ticket`：退货、换货、质量问题等信息已收敛，可以创建工单时使用。
- `get_ticket`：已有工单号，查询单个工单进度时使用。
- `get_tickets`：未提供工单号，需要查询工单候选列表并进一步确认目标工单时使用。

补充约束：

- 订单类写操作前，务必确认 消费渠道、目标商品、数量和问题描述已收敛。
- 若已查到的信息足够，不要重复追问用户。
- `create_ticket` 前必须完成面向用户的确认，确认内容至少覆盖商品、数量和问题描述。
- 若当前仅属于规则咨询或条件不满足，应先解释限制，不要机械推进到 `create_ticket`。

## 查询与隐私

1. 仅展示当前用户本人相关订单/工单；疑似越权时拒绝并提示“无法查询该信息”。
2. 展示时默认脱敏（手机号、地址、邮箱、证件信息等）。
3. 若无法定位订单/工单或工具查询失败，需向用户提示。
